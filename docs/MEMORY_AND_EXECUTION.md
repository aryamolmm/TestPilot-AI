# TestPilot AI: Memory & Execution Architecture

This document provides a technical overview of how TestPilot AI persists test intelligence and the recommended local development workflow.

## Memory Storage

TestPilot AI uses a decentralized persistence layer to ensure high-velocity access to past test intelligence without the overhead of a heavy database.

*   **Persistence Layer:** All execution history is stored in a local `memory.json` file at the project root.
*   **Data Captured:** For every successful run, the system persists:
    *   **User Input:** The original feature description or Jira requirement string.
    *   **Architect Output:** The synthetically generated Gherkin/BDD scenarios.
    *   **Automation Output:** The ready-to-run Playwright TypeScript test scripts.
    *   **Quality Report:** Coverage analysis results, including quality scores and identified gaps.
*   **Storage Trigger:** Memory entries are committed atomicly **after the final agent (ReworkAgent)** has finalized the test suite. This ensures that the memory index only contains validated, high-quality test assets.

## Memory Retrieval

The **MemoryAgent** acts as the gatekeeper for every new request, scanning the history to identify optimization opportunities.

*   **Scanning Process:** When a new requirement is submitted, the system reads the `memory.json` index and performs a semantic comparison.
*   **Similarity Logic:** Retrieval is based on a **textual similarity algorithm** that calculates the overlap between the new input and stored historical inputs.
*   **Decision Logic:** 
    *   **High Similarity (>85%)**: The system identifies a direct match and offers to **Reuse** the existing verified assets, bypassing new generation time.
    *   **Moderate Similarity (45% - 85%)**: The system identifies an evolution. It retrieves the memory and triggers **Improvement Mode**, where the ReworkAgent uses the past code as a baseline.
    *   **Low Similarity (<45%)**: The system ignores memory and initiates a **Fresh Generation** pipeline.

## Memory Usage in Pipeline

Memory transforms TestPilot AI from a static generator into an **adaptive intelligence system**.

*   **Output Reuse:** Significant reduction in token usage and execution time by serving previously validated Gherkin and Playwright scripts.
*   **Iterative Improvement:** Instead of rewriting tests for every small change, the ReworkAgent uses memory to understand the "previous state" and only adds or modifies what is necessary.
*   **Adaptive Decision Making:** Memory influences the Orchestrator's path. If a specific coverage gap was previously identified and fixed, the system "remembers" that refinement, ensuring future related tests are robust from the start.

## Local Execution Flow

Follow these steps to set up and observe the autonomous pipeline on your local machine.

### 1. Prerequisite Installation
Ensure you have Node.js installed, then install all project dependencies:
```bash
npm install
```

### 2. Environment Configuration
Create a `.env` file in the root directory and provide your API credentials:
```env
GROQ_API_KEY=your_key_here
JIRA_API_TOKEN=your_token_here
JIRA_EMAIL=your_email_here
JIRA_URL=https://your-domain.atlassian.net
```

### 3. Starting the System
Open two terminal windows to run the dedicated services:
*   **Terminal 1 (Backend API):** `node api/index.js` (Runs at http://localhost:3001)
*   **Terminal 2 (Frontend UI):** `npm run dev` (Runs at http://localhost:5173)

### 4. Running the Super Agent
Navigate to the **Super Agent** tab in the UI:
1.  **First Run (Discovery):** Provide a new requirement (e.g., "Verify Swag Labs Checkout"). The system logs "Memory Scan: No Match" and executes the full four-agent pipeline.
2.  **Second Run (Optimization):** Provide a similar requirement. Observe the **MemoryAgent** in the execution trace identifying the match and accelerating the pipeline through reuse or targeted improvement.
